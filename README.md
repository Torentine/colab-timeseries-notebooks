# Отчет об исследовании временного ряда

## Описание временного ряда

В работе используется датасет `Electricity Consumption and Production Data` с почасовыми измерениями потребления и производства электроэнергии. Для решения задачи прогнозирования исходный ряд был агрегирован до дневного среднего значения целевой переменной `Consumption`.

Подготовленный временной ряд:

| Параметр | Значение |
| --- | --- |
| Целевая переменная | `y`, среднесуточное потребление электроэнергии |
| Частота | 1 день |
| Период | 2019-01-01 - 2024-03-31 |
| Количество наблюдений | 1917 |
| Основной ноутбук | `02_electricity_forecasting_colab.ipynb` |
| Данные | `data/electricity_daily_prepared.csv` |

Дополнительно в репозитории есть ноутбук `01_etth1_time_series_colab.ipynb` для анализа ряда ETTh1. Основной отчет ниже относится к задаче прогнозирования потребления электроэнергии.

## Постановка задачи

Цель исследования: построить воспроизводимый пайплайн анализа временного ряда и получить прогноз среднесуточного потребления электроэнергии на 7 дней вперед.

Критерии качества:

- `MAE` - средняя абсолютная ошибка;
- `RMSE` - корень из средней квадратичной ошибки;
- `MAPE` - средняя абсолютная процентная ошибка.

Тестовая выборка: последние 7 дней ряда. Дополнительно выполнено rolling backtesting для проверки устойчивости пайплайна на нескольких временных окнах.

## EDA временного ряда

На этапе разведочного анализа были проверены:

- типы данных и корректность временной метки;
- пропуски и дубликаты дат;
- регулярность дневной частоты;
- месячная, недельная и годовая сезонность;
- автокорреляция на ключевых лагах;
- необходимость лаговых и календарных признаков.

Автокорреляция показала сильную зависимость от предыдущих значений и выраженную недельную сезонность:

| Лаг | Автокорреляция |
| --- | ---: |
| 1 | 0.792 |
| 2 | 0.576 |
| 3 | 0.502 |
| 7 | 0.854 |
| 14 | 0.762 |
| 28 | 0.712 |
| 365 | 0.563 |

Выводы EDA:

- лаги `1`, `7`, `14`, `28` оправданы высокой автокорреляцией;
- недельная сезонность является ключевой, поэтому для статистических моделей выбран сезонный период `7`;
- календарные признаки `dayofweek` и `month` полезны для ML/DL-моделей;
- rolling-окна `7`, `14`, `28` отражают краткосрочный и среднесрочный уровень ряда.

## Анализ аномалий

Сравнивались три метода поиска аномалий:

| Метод | Параметры | Количество аномалий | Комментарий |
| --- | --- | ---: | --- |
| Rolling z-score | `window=30`, `threshold=3` | 3 | Находит только резкие локальные выбросы. Метод простой, но плохо учитывает сезонность. |
| Seasonal residual IQR | `period=7`, `multiplier=1.5` | 93 | Сначала удаляет недельную сезонность через STL, затем ищет выбросы в остатках. Лучше подходит для данного ряда. |
| Feature distance detector | `contamination=0.02` | 38 | Использует признаки ряда, но параметр доли выбросов задается заранее и менее интерпретируем. |

Основным выбран метод `Seasonal residual IQR`, потому что EDA показал сильную недельную сезонность. Если искать выбросы напрямую по исходному ряду, нормальные сезонные пики могут ошибочно считаться аномалиями. Поэтому сначала ряд раскладывается на сезонную компоненту и остаток, а затем правило IQR применяется к остаткам.

Параметры метода:

- `period=7`, потому что ряд дневной и имеет недельную сезонность;
- `multiplier=1.5`, потому что это стандартное IQR-правило для умеренно чувствительного поиска выбросов.

- DL-модели проверены через несколько MLP-конфигураций, но не превзошли линейные ML-модели.

## Пайплайн решения задачи

Итоговый пайплайн состоит из следующих шагов:

1. Загрузка данных из `data/electricity_daily_prepared.csv`.
2. Проверка временной метки, сортировка, удаление дублей.
3. Формирование дневного ряда `ds`, `y`.
4. Генерация признаков:
   - лаги `1`, `2`, `3`, `7`, `14`, `28`;
   - rolling mean и rolling std для окон `7`, `14`, `28`;
   - календарные признаки `dayofweek`, `month`.
5. Разделение на train/test, где test - последние 7 дней.
6. Обучение модели `Ridge(alpha=25)`.
7. Рекурсивный прогноз на 7 дней вперед.
8. Расчет `MAE`, `RMSE`, `MAPE`.
9. Rolling backtesting для проверки устойчивости на нескольких окнах.

Выбор `Ridge(alpha=25)` обоснован тем, что признаки содержат несколько взаимосвязанных лагов и rolling-статистик. Регуляризация снижает риск нестабильных коэффициентов и делает пайплайн более устойчивым.
